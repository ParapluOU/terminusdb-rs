# terminusdb-orm

An ActiveRecord-style ORM layer for TerminusDB with compile-time type safety and efficient batch loading of related entities.

## Features

- **No hand-written GraphQL**: queries are expressed as Rust — typed filters,
  ordering, and relation traversal — and the ORM renders the GraphQL for you.
  You should rarely (if ever) need to write a GraphQL filter string by hand.
- **Type-safe queries**: `EntityIDFor<T>` ensures you can only query with IDs of the correct type
- **Derive-generated typed filters**: every model gets a `{Model}Filter` (and
  `{Model}Ordering`) struct with a field per model field, so filters are checked
  by the compiler — see [Filtering](#filtering-with-modelfilter).
- **Type-directed GraphQL rendering** (`ToGql`): strings are quoted, enums are
  bare, ids/dates quoted, nested filters recursive — no enum-vs-string guessing.
- **Automatic relation detection**: Derive macro generates relation traits from `TdbLazy<T>` and `EntityIDFor<T>` fields
- **Compile-time relation validation**: Invalid relations fail at compile time, not runtime
- **Efficient batch loading**: Always exactly 2 database calls regardless of query complexity (no N+1)
- **Order-preserving**: an ordered query's sort survives the Phase-2 batch fetch
- **Typed results**: `get_ordered` / `get_indexed` hand back `EntityIDFor<T>`,
  not stringly ids — the canonical `@id` is parsed once, at the boundary
- **GraphQL-based relation traversal**: Uses TerminusDB's auto-generated reverse fields

## Quick Start

```rust
use terminusdb_orm::prelude::*;
use terminusdb_schema::TdbLazy;
use terminusdb_schema_derive::TerminusDBModel;
use serde::{Serialize, Deserialize};

// Define models with relations
#[derive(Clone, Debug, Serialize, Deserialize, TerminusDBModel)]
pub struct Writer {
    pub name: String,
}

#[derive(Clone, Debug, Serialize, Deserialize, TerminusDBModel)]
pub struct BlogPost {
    pub title: String,
    /// TdbLazy creates a document link, enabling reverse relations
    pub writer: TdbLazy<Writer>,
}

// Query with eager loading
let writer_id = EntityIDFor::<Writer>::new("writer1")?;
let result = Writer::find(writer_id)
    .with::<BlogPost>()  // Load all BlogPosts by this writer
    .with_client(&client)
    .execute(&spec)
    .await?;

// Access results by type
let writers: Vec<Writer> = result.get()?;
let posts: Vec<BlogPost> = result.get()?;
```

## Defining Relations

### Document Links with `TdbLazy<T>`

`TdbLazy<T>` creates an actual document link in the TerminusDB schema. This enables:
- Forward traversal (BlogPost -> Writer)
- Reverse traversal via auto-generated fields (Writer -> all BlogPosts)

```rust
#[derive(TerminusDBModel)]
pub struct BlogPost {
    pub title: String,
    pub writer: TdbLazy<Writer>,  // Document link
}
```

### ID References with `EntityIDFor<T>`

`EntityIDFor<T>` stores a string ID reference. It provides compile-time type safety for IDs but **does not create schema links** - the field is stored as a plain string in TerminusDB:

```rust
#[derive(TerminusDBModel)]
pub struct Comment {
    pub text: String,
    pub post_id: EntityIDFor<Post>,    // Stored as string, no GraphQL traversal
    pub author_id: EntityIDFor<User>,  // Stored as string, no GraphQL traversal
}
```

> **When to use each**:
> - Use `TdbLazy<T>` when you need relation traversal (`.with()`, `.with_field()`)
> - Use `EntityIDFor<T>` when you only need type-safe ID storage/validation

### Auto-Generated Traits

The `TerminusDBModel` derive macro automatically generates:

- `{Struct}Fields` module with marker types for each field
- `ReverseRelation<T, Field>` for each `TdbLazy<T>` or `EntityIDFor<T>` field
- `ForwardRelation<T, Field>` for forward traversal
- `ReverseRelation<T, DefaultField>` for `.with::<T>()` without specifying a field

## Query API

### Finding Entities

```rust
// Find by typed ID
let id = EntityIDFor::<Comment>::new("comment1")?;
let query = Comment::find(id);

// Find multiple by typed IDs
let ids = vec![
    EntityIDFor::<Comment>::new("1")?,
    EntityIDFor::<Comment>::new("2")?,
];
let query = Comment::find_all(ids);

// Find by string ID (bypasses type checking)
let query = Comment::find_by_string("Comment/1");
let query = Comment::find_all_by_strings(["Comment/1", "Comment/2"]);
```

### Filtering with `{Model}Filter`

`Model::query(filter)` runs a filtered top-level query. The `{Model}Filter`
type is **generated by the `TerminusDBModel` derive** (see
[Generating filters](#generating-filters)) — one optional field per model
field, so the whole filter is type-checked:

- **scalar fields** → a scalar filter (`StringFilter`, `IntFilter`,
  `BigIntFilter`, `FloatFilter`, `DateTimeFilter`, `BooleanFilter`), each with
  `eq` / `ne` / `lt` / `le` / `gt` / `ge` (+ `regex` / `startsWith` /
  `allOfTerms` / `anyOfTerms` on strings)
- **relation fields** (`TdbLazy<T>` / `EntityIDFor<T>`) → the target's
  `{Target}Filter`, **boxed** (nested — filter by a field on the linked doc)
- **enum fields** → `{Enum}Filter { eq, ne }` (enum values render bare)
- an envelope, with `tdb_`-prefixed Rust names to dodge collisions with fields
  literally named `id`/`and`/…: `tdb_id` (`_id`, a typed `EntityIDFor<Model>`),
  `tdb_ids` (`_ids`), `tdb_and` / `tdb_or` (`Vec<Box<Self>>`), `tdb_not`

```rust
use terminusdb_schema::{StringFilter, IntFilter};

// Generated for: struct BlogPost { title: String, writer: TdbLazy<Writer>, views: u32 }
let result = BlogPost::query(BlogPostFilter {
        title: Some(StringFilter { eq: Some("Hello".into()), ..Default::default() }),
        // nested relation filter: BlogPosts whose Writer is named "Ada"
        writer: Some(Box::new(WriterFilter {
            name: Some(StringFilter { eq: Some("Ada".into()), ..Default::default() }),
            ..Default::default()
        })),
        views: Some(IntFilter { ge: Some(100), ..Default::default() }),
        ..Default::default()
    })
    .limit(20)
    .offset(0)
    .with_client(&client)
    .execute(&spec).await?;
let posts: Vec<BlogPost> = result.get()?;
```

Boolean combinators via the envelope:

```rust
// title == "A" OR title == "B"
let by_title = |t: &str| BlogPostFilter {
    title: Some(StringFilter { eq: Some(t.into()), ..Default::default() }),
    ..Default::default()
};
BlogPost::query(BlogPostFilter {
    tdb_or: Some(vec![Box::new(by_title("A")), Box::new(by_title("B"))]),
    ..Default::default()
});

// filter by id(s) — typed, rendered as the full IRI TDB's `_id`/`_ids` expect
BlogPost::query(BlogPostFilter {
    tdb_id: Some(EntityIDFor::<BlogPost>::new("BlogPost/1")?),
    ..Default::default()
});
```

> **No hand-written GraphQL.** The filter is rendered to a GraphQL filter
> object by the derive-generated `ToGql` impl: `String`/`DateTime`/`EntityIDFor`
> are quoted, numbers/bools/enums are bare, lists and nested filters recurse.
> (This replaced an older `serde_json` + lowercase heuristic that wrongly left
> lowercase string values unquoted.)

### Ordering with `{Model}Ordering`

`{Model}Ordering` is generated alongside the filter, with a field per
**orderable scalar** (strings, numbers, bool, `DateTime` — not relations or
enums, mirroring TDB's `{Model}_Ordering`). Each field takes a
`TerminusOrdering` (`Asc` / `Desc`); the Phase-2 fetch preserves the order.

```rust
use terminusdb_schema::TerminusOrdering;

let result = BlogPost::query(BlogPostFilter::default())
    .order_by(BlogPostOrdering {
        created_at: Some(TerminusOrdering::Desc),
        ..Default::default()
    })
    .with_client(&client)
    .execute(&spec).await?;
let newest_first: Vec<BlogPost> = result.get()?; // sort preserved through the batch fetch
```

### Generating filters

`{Model}Filter` / `{Model}Ordering` (and their `ToGql` impls) are emitted by the
`TerminusDBModel` derive, **opt-out**: a consuming crate turns them on by setting
`TERMINUSDB_DERIVE_FILTERS=1` for its own compilation (in its `build.rs`), and an
individual model opts out with `#[tdb(no_filter)]`.

```rust
// build.rs of the crate that defines your models
fn main() {
    println!("cargo:rustc-env=TERMINUSDB_DERIVE_FILTERS=1");
}
```

It's a build-time env var rather than a Cargo feature on purpose: a feature
unifies across the whole dependency graph and would force filter generation onto
unrelated model crates (whose exotic field types don't all produce clean
filters); the env var scopes cleanly to the opting-in crate.

### Loading Relations

#### Reverse Relations: `.with::<T>()`

Load entities that reference the queried entity:

```rust
// BlogPost has: writer: TdbLazy<Writer>
// Load Writer and all BlogPosts that reference it
let result = Writer::find(id)
    .with::<BlogPost>()
    .execute(&spec).await?;
```

#### Reverse Relations with Field: `.with_via::<T, Field>()`

When a type has multiple fields referencing the same target, specify which one:

```rust
#[derive(TerminusDBModel)]
pub struct Document {
    pub author: TdbLazy<User>,    // User who wrote it
    pub reviewer: TdbLazy<User>,  // User who reviewed it
}

// Load only documents where user is the author
let result = User::find(id)
    .with_via::<Document, DocumentFields::Author>()
    .execute(&spec).await?;

// Load only documents where user is the reviewer
let result = User::find(id)
    .with_via::<Document, DocumentFields::Reviewer>()
    .execute(&spec).await?;
```

#### Forward Relations: `.with_field::<T, Field>()`

Load entities that the queried entity references. **Requires `TdbLazy<T>` fields** (not `EntityIDFor<T>`):

```rust
#[derive(TerminusDBModel)]
pub struct Car {
    pub front_left: TdbLazy<Wheel>,   // Document link - forward traversal works
    pub front_right: TdbLazy<Wheel>,
    pub back_left: TdbLazy<Wheel>,
    pub back_right: TdbLazy<Wheel>,
}

// Load all wheels for a car
let result = Car::find(id)
    .with_field::<Wheel, CarFields::FrontLeft>()
    .with_field::<Wheel, CarFields::FrontRight>()
    .with_field::<Wheel, CarFields::BackLeft>()
    .with_field::<Wheel, CarFields::BackRight>()
    .execute(&spec).await?;
```

> **Note**: Forward relations with `EntityIDFor<T>` fields will compile but fail at runtime because `EntityIDFor` creates string fields, not document links. GraphQL cannot traverse into strings.

#### Nested Relations: `.with_nested::<T>()`

Load relations of relations:

```rust
let result = Writer::find(id)
    .with_nested::<BlogPost>(|b| {
        b.with::<Comment>()  // Comments on each BlogPost
         .with::<Like>()     // Likes on each BlogPost
    })
    .execute(&spec).await?;
```

#### Filtered / ordered / paginated relations: `.with_opts::<R, F, O>(RelationOpts)`

Apply a filter, ordering, and/or limit/offset to a loaded relation.
`RelationOpts::new()` starts empty (`F = O = ()` — the no-op filter/ordering),
and `.filter(..)` / `.order_by(..)` set the concrete types:

```rust
use terminusdb_orm::RelationOpts;
use terminusdb_schema::{BooleanFilter, TerminusOrdering};

// A Writer's 5 most-recent *published* BlogPosts
let result = Writer::find(id)
    .with_opts::<BlogPost, BlogPostFilter, BlogPostOrdering>(
        RelationOpts::new()
            .filter(BlogPostFilter {
                published: Some(BooleanFilter { eq: Some(true), ..Default::default() }),
                ..Default::default()
            })
            .order_by(BlogPostOrdering {
                created_at: Some(TerminusOrdering::Desc),
                ..Default::default()
            })
            .limit(5))
    .with_client(&client)
    .execute(&spec).await?;

// Ordering only (no filter) — leave F as the inferred `()`:
Writer::find(id).with_opts::<BlogPost, (), BlogPostOrdering>(
    RelationOpts::new().order_by(BlogPostOrdering {
        created_at: Some(TerminusOrdering::Asc), ..Default::default()
    }));
```

### Executing Queries

```rust
// Execute and get OrmResult containing all types
let result = query.execute(&spec).await?;

// Execute and get only the primary type
let writers: Vec<Writer> = query.execute_primary(&spec).await?;

// Execute and get exactly one result (errors if 0 or >1)
let writer: Writer = query.execute_one(&spec).await?;
```

### Accessing Results

```rust
let result = Writer::find(id)
    .with::<BlogPost>()
    .execute(&spec).await?;

// Get all entities of a type
let writers: Vec<Writer> = result.get()?;
let posts: Vec<BlogPost> = result.get()?;

// Get a single entity (errors if >1)
let writer: Option<Writer> = result.get_one()?;

// (id, model) pairs — id is a typed EntityIDFor<T>, parsed once from the
// canonical @id. `get_ordered` keeps document order (i.e. an ordered query's
// sort); `get_indexed` keys a HashMap by id.
use std::collections::HashMap;
let ordered: Vec<(EntityIDFor<BlogPost>, BlogPost)> = result.get_ordered::<BlogPost>()?;
let by_id: HashMap<EntityIDFor<BlogPost>, BlogPost> = result.get_indexed::<BlogPost>()?;

// Metadata
println!("Total docs: {}", result.len());
println!("Types: {:?}", result.class_names());
println!("Counts: {:?}", result.count_by_class());
```

## How It Works

The ORM uses a **two-phase loading** strategy that always results in exactly 2 database calls:

1. **Phase 1: GraphQL ID Collection**
   - Generates a GraphQL query that traverses all requested relations
   - Uses TerminusDB's auto-generated reverse fields (e.g., `_writer_of_BlogPost`)
   - Collects only `_id` values, not full documents

   - Filters and ordering are rendered into this GraphQL by the
     derive-generated `ToGql` impls (type-directed — no string templating)

2. **Phase 2: Batch Document Fetch**
   - Fetches all collected IDs in a single batch request
   - Returns documents of all types in one response
   - **Order-preserving**: the Phase-1 id order (e.g. from `order_by`) is kept
     through the batch fetch, so `get`/`get_ordered` come back sorted

Example generated GraphQL for `Writer::find(id).with::<BlogPost>()`:

```graphql
query {
  Writer(id: "terminusdb:///data/Writer/123") {
    _id
    _writer_of_BlogPost {
      _id
    }
  }
}
```

## Compile-Time Safety

Invalid relations fail at compile time:

```rust
// This compiles: BlogPost has TdbLazy<Writer>
Writer::find(id).with::<BlogPost>();

// This does NOT compile: Writer has no TdbLazy<BlogPost>
BlogPost::find(id).with::<Writer>();  // Error: Writer: ReverseRelation<BlogPost> not satisfied

// This compiles: Car has EntityIDFor<Wheel> fields
Car::find(id).with_field::<Wheel, CarFields::FrontLeft>();

// This does NOT compile: wrong field type
Car::find(id).with_field::<Wheel, UserFields::Name>();  // Error: type mismatch
```

## Testing

The ORM includes a convenient `with_test_db` helper for integration tests:

```rust
use terminusdb_orm::testing::with_test_db;

#[tokio::test]
async fn test_something() -> anyhow::Result<()> {
    with_test_db("my_test", |client, spec| async move {
        // Insert schema
        client.insert_entity_schema::<MyModel>((&spec).into()).await?;

        // Insert data, run queries...

        Ok(())
    }).await
}
```

This automatically:
1. Gets or creates a shared in-memory test server
2. Creates a uniquely-named database
3. Runs your test closure
4. Cleans up the database when done (even on failure)

## Current Limitations

### Implemented since the early versions of this README

- **Filtered / ordered / paginated queries and relations** — via
  `Model::query(filter).order_by(..).limit(..)` and
  `.with_opts::<R, F, O>(RelationOpts::new().filter(..).order_by(..).limit(..))`
  (see [Filtering](#filtering-with-modelfilter) / [Ordering](#ordering-with-modelordering)).

### Not Yet Implemented

- **Lazy loading**: On-demand relation loading when accessed
- **Collection-membership filters**: filtering a `Vec<TdbLazy<T>>` field by a
  related field (e.g. assets having a tag named X) — filter such cases on the
  fetched result for now
- **`u64` beyond `i64::MAX`** in `BigIntFilter` (stored as `i64`)

### Known Constraints

- **`TdbLazy<T>` is required for relation traversal** - both forward (`.with_field`) and reverse (`.with`) relations rely on GraphQL traversal, which requires actual document links in the schema
- `EntityIDFor<T>` fields store string IDs only - useful for type-safe ID handling but cannot be traversed via GraphQL
- For `EntityIDFor<T>` fields, you must manually read the ID and fetch the related document separately

## Running Tests

```bash
# Run all ORM tests (uses embedded in-memory TerminusDB)
cargo test -p terminusdb-orm --features testing

# Run a specific test
cargo test -p terminusdb-orm --features testing --test typed_api_test
```
